## 一、项目概述

### 1.1 项目背景

1553B总线是一种广泛应用于航空航天领域的串行数据总线标准。本工具旨在为工程师提供一套完整的1553B总线数据分析解决方案，涵盖数据加载、解析、展示、筛选、可视化、AI智能分析、报告生成等全流程功能。

### 1.2 系统目标

1. 提供高效的1553B二进制数据解析能力，支持大小端字节序
2. 提供多维度的数据筛选和统计功能
3. 集成AI大模型实现自然语言查询和智能分析
4. 生成专业的数据分析报告（PDF/DOCX/HTML）
5. 支持大数据量场景下的数据库存储模式

### 1.3 用户角色

| 角色 | 描述 |
|------|------|
| 数据分析师 | 加载和分析1553B总线数据，生成报告 |
| 系统工程师 | 使用AI查询功能进行故障诊断和异常检测 |
| 测试工程师 | 使用测试数据生成功能验证系统行为 |

## 二、功能需求

### 2.1 数据加载与解析（M-01）

#### F-001 标准二进制文件加载
- 支持打开 .raw / .bin 格式的二进制文件
- 文件包头标识：0xA5A5 0xA5
- 数据包头标识：0xAABB
- 加载过程显示进度条（0%→100%）和耗时计时器
- 解析完成后弹框提示成功，显示解析数量和耗时

#### F-002 批量多文件加载
- 支持 Ctrl+点击选择多个文件
- 支持拖拽多个文件到窗口
- 文件列表面板显示所有已加载文件

#### F-003 文件夹批量导入
- 支持选择文件夹，自动扫描目录下所有二进制文件
- 逐个导入并更新文件列表
- 显示总体进度和单个文件进度

#### F-004 无效文件格式处理
- 识别非二进制格式文件
- 弹出友好错误提示："文件格式不支持"
- 不崩溃，界面保持可用
- 记录 ERROR 级别日志

#### F-005 取消加载操作
- 加载过程中可点击取消按钮
- 加载立即停止，进度条关闭
- 已加载的部分数据被清理，不残留未完成数据

#### F-006 子线程加载不阻塞UI
- 文件解析在子线程执行（QThread + moveToThread）
- 加载过程中 UI 保持可交互
- 无卡顿或假死现象

#### F-008 进度实时反馈
- 进度条平滑更新
- 状态文本实时更新（"正在解析数据..."等）
- 耗时计时器 HH:MM:SS 格式实时显示

#### F-011 命令字位域解析
- 终端地址 (bit 15-11)：0-31
- T/R 位 (bit 10)：0=BC→RT, 1=RT→BC
- 子地址 (bit 9-5)：0-31
- 数据计数/方式码 (bit 4-0)：0-31

#### F-013 时间戳转换
- 时间戳 = timestamp × 40μs，转换为毫秒显示
- 显示格式：HH:MM:SS.mmm
- 使用 double 计算避免浮点截断误差

#### F-015 消息类型自动识别
- T_R=0 且 cmd2无效 → BC→RT
- T_R=1 → RT→BC
- cmd1和cmd2都有效且地址不同 → RT→RT
- 终端地址=31 → Broadcast

#### F-020 包头校验
- header1 ≠ 0xA5A5 的帧标记为错误（error_flag = true）
- 错误统计中显示包头校验失败数量
- 表格中错误行高亮显示

#### F-021 数据头校验
- data header ≠ 0xAABB 的帧标记为错误

#### F-022 包长度校验
- packetLen 与实际长度不一致的帧标记为错误
- 不影响后续数据的解析

#### F-024 16进制查看器
- 十六进制以空格分隔显示（如 "A5 A5 A5 ..."）
- ASCII 字符并行显示
- 支持分页浏览大文件
- 字节序设置正确生效

#### F-085 字节序支持
- 支持大端（Big Endian）和小端（Little Endian）字节序
- 可通过配置切换字节序模式

#### F-NEW-001 数据区长度计算规则
- 子地址 = 0 或 31：datas 长度固定为 2 字节
- 子地址非 0/31 且数据计数 = 0：datas 长度为 64 字节
- 子地址非 0/31 且数据计数 > 0：datas 长度 = 数据计数 × 2 字节

### 2.7 智能分析报告（M-07）

#### F-090 报告生成流程
- 四阶段异步生成：
  1. 收集统计数据（单次遍历）
  2. 生成5张图表（饼图/柱状图/折线图/错误分布/成功率）
  3. 并行AI分析（7个独立任务）
  4. 导出报告文件
- 支持取消报告生成

#### F-091 报告章节结构
- 第一章：1553B协议概述
- 第二章：数据统计分析
- 第三章：数据可视化图表
- 第四章：AI智能分析
- 第五章：结论与建议

#### F-092 报告统计数据
- 总消息数、各类型消息数及占比
- 终端通信量统计
- 时间范围
- 错误率和成功率

#### F-093 AI分析内容
- 数据概览分析
- 通信模式分析
- 异常检测分析
- 终端行为分析
- 时间特征分析
- 错误模式分析
- 综合建议

#### F-NEW-007 PDF报告排版规范
- **页面设置**：A4纵向，300 DPI，页边距25mm
- **字体规范**：
  - 一级标题：黑体22pt，居中加粗
  - 二级标题：黑体16pt，左对齐加粗，底部边框
  - 三级标题：宋体14pt，左对齐加粗
  - 四级标题：宋体12pt，左对齐不加粗
  - 正文：宋体9pt，首行缩进2字符（18pt），两端对齐
  - 表头：宋体9pt，加粗，灰色背景(#D9D9D9)，居中
  - 表格单元格：宋体7.5pt，居中
  - 表名/图名：宋体7.5pt，加粗，居中
- **表格规范**：
  - 所有表格宽度100%，居中显示
  - 外边框1.5px，内边框1px，黑色实线
- **图片规范**：
  - 图表宽度为页面内容宽度的112.5%（原75%放大50%），封顶98%
  - 图片前强制换行，与上方标题保持间距
  - 使用2倍分辨率渲染保证打印清晰度
- **段落规范**：
  - 正文首行缩进2字符（QTextCursor API设置，非CSS）
  - 图片块上方间距12pt
  - AI分析区域左侧蓝色边框(3pt #3498DB)

#### F-NEW-008 DOCX报告排版规范
- 与PDF报告相同的章节结构和内容
- 手动构建OOXML ZIP包（无第三方库依赖）

## 八、未实现功能详细设计方案

### 8.1 Bug修复类

#### 8.1.1 F-NEW-012: DataStore异步筛选线程安全问题

**问题描述：**
DataStore类中`applyFiltersAsync()`使用`QtConcurrent::run`在后台线程执行筛选，但DataStore的成员变量（如`m_allRecords`、`m_filteredRecords`、`m_currentPageRecords`等）没有任何互斥锁保护。当后台线程正在读取`m_allRecords`进行筛选时，UI线程可能同时修改这些数据（如加载新文件、添加新数据），导致数据竞争和未定义行为。

**设计方案：**
```
1. 在DataStore类中添加QReadWriteLock成员：
   QReadWriteLock m_dataLock;

2. 读操作使用读锁（QReadLocker）：
   - applyFiltersAsync()中拷贝m_allRecords前加读锁
   - getCurrentPageRecords()、getFilteredRecords()等查询方法加读锁
   - getTerminalStatistics()等统计方法加读锁

3. 写操作使用写锁（QWriteLocker）：
   - addRecord()、loadFromBinaryFile()等修改方法加写锁
   - applyFilters()修改m_filteredRecords时加写锁
   - clearAll()、clearFilter()等清空方法加写锁

4. 异步筛选特殊处理：
   - applyFiltersAsync()入口处拷贝m_allRecords快照（加读锁）
   - 后台线程只操作快照数据，不访问成员变量
   - 结果回写时通过信号在UI线程执行（加写锁）

5. 注意事项：
   - 避免在持有锁时发送信号（可能导致死锁）
   - 锁粒度尽量小，避免长时间持锁阻塞UI
   - 读写锁优于互斥锁，允许多个并发读操作
```

**涉及文件：** DataStore.h、DataStore.cpp

---

#### 8.1.2 F-NEW-013: 同步/异步筛选竞态条件

**问题描述：**
`applyFilters()`（同步）和`applyFiltersAsync()`（异步）可能被并发调用。例如用户快速切换筛选条件时，前一个异步筛选尚未完成，新的同步筛选就开始执行，导致结果不一致。`m_cancelFilter`标志位也存在竞态条件。

**设计方案：**
```
1. 添加筛选状态枚举：
   enum class FilterState { Idle, Filtering, AsyncFiltering };
   QAtomicInt m_filterState;  // 原子变量，线程安全

2. applyFilters()入口检查：
   - 如果当前处于AsyncFiltering状态，先设置m_cancelFilter=true
   - 等待异步筛选结束（QEventLoop或定时轮询，超时500ms）
   - 超时则强制继续（异步筛选结果会被覆盖）

3. applyFiltersAsync()入口检查：
   - 如果当前处于Filtering状态，直接返回（不允许并发异步筛选）
   - 设置状态为AsyncFiltering

4. 筛选完成时重置状态为Idle

5. 替代方案（推荐）：
   - 统一使用异步筛选，applyFilters()内部调用applyFiltersAsync()
   - 通过QFuture/QFutureWatcher跟踪筛选任务
   - 新筛选请求自动取消前一个未完成的筛选
   - 使用QThreadPool::cancel()或自定义取消机制
```

**涉及文件：** DataStore.h、DataStore.cpp

---

#### 8.1.3 F-NEW-014: ExportService PDF导出无行数上限但大数据量内存溢出

**问题描述：**
ExportService::exportToPdf()调用`store->getRecordsByScope(scope)`一次性获取所有数据记录。当数据量极大（如100万条）时，QVector<DataRecord>会占用大量内存，且生成的HTML文档也会非常大，可能导致内存溢出或程序崩溃。

**设计方案：**
```
1. 分批导出策略：
   - 添加分页导出参数，每批处理pageSize条记录（默认5000条）
   - 使用QTextDocument分批追加内容
   - 避免一次性构建完整HTML字符串

2. 数据库模式优化：
   - 当DataStore处于数据库模式时，使用分页查询
   - 循环调用DatabaseManager::queryPackets()按页获取数据
   - 每批数据处理完后立即写入文档，释放内存

3. 内存模式优化：
   - 使用迭代器模式，不一次性拷贝所有记录
   - 每处理一批记录后清空临时变量

4. 进度反馈：
   - 每处理一批记录后发送exportProgress信号
   - 支持取消操作（检查m_cancelExport标志）

5. 大数据量保护：
   - 导出前检查记录总数，超过10万条时弹框确认
   - 超过50万条时建议用户先筛选再导出
```

**涉及文件：** ExportService.h、ExportService.cpp

#### 8.1.4 F-NEW-015: ExportService PDF导出字段不完整

**设计方案：**

#### 8.3.3 F-NEW-023: DataStore筛选性能优化（CMD缓存+位运算）

**问题描述：**
DataStore::matchesFilter()中对每条记录都进行CMD位域解析（memcpy到CMD结构体），重复解析开销大。当数据量达到百万级时，筛选性能显著下降。

**设计方案：**
```
1. DataRecord中缓存CMD解析结果：
   struct DataRecord {
       quint16 cmd1Raw;       // 原始CMD1命令字
       quint8  terminalAddr;  // 已有：终端地址
       quint8  subAddr;       // 已有：子地址
       quint8  t_r;           // 已有：T/R位
       quint8  dataCount;     // 已有：数据计数
       
       quint16 cmd2Raw;       // 新增：原始CMD2命令字
       quint8  terminalAddr2; // 新增：终端地址2
       quint8  subAddr2;      // 新增：子地址2
       quint8  t_r2;          // 新增：T/R位2
       quint8  dataCount2;    // 新增：数据计数2
   };

2. 使用位运算替代memcpy+结构体：
   // 旧方式（慢）：
   CMD cmd; memcpy(&cmd, &record.packetData.cmd1, sizeof(CMD));
   int addr = cmd.zhongduandizhi;
   
   // 新方式（快）：
   int addr = (record.packetData.cmd1 >> 11) & 0x1F;  // bit15-11

3. 位运算宏定义：
   #define CMD_TERMINAL_ADDR(cmd)  (((cmd) >> 11) & 0x1F)
   #define CMD_SUB_ADDR(cmd)       (((cmd) >> 5) & 0x1F)
   #define CMD_TR_BIT(cmd)         (((cmd) >> 10) & 0x1)
   #define CMD_WORD_COUNT(cmd)     ((cmd) & 0x1F)

4. 筛选时直接使用缓存字段，避免重复解析

5. 涉及文件：
   - 修改：src/core/datastore/DataStore.h（DataRecord结构体）
   - 修改：src/core/datastore/DataStore.cpp（matchesFilter方法）
   - 修改：src/core/parser/PacketParser.h（添加位运算宏）
```

#### 8.3.6 F-NEW-026: 数据库查询缓存和预加载

**问题描述：**
数据库模式下每次翻页都需要执行SQL查询，频繁的磁盘I/O导致翻页延迟明显。

**设计方案：**
```
1. 添加LRU缓存层：
   class DbQueryCache {
   public:
       // 缓存键：fileId + 筛选条件hash + 页码
       struct CacheKey {
           int fileId;
           quint32 filterHash;
           int page;
           bool operator==(const CacheKey& other) const;
       };
       
       // 缓存值：查询结果记录列表
       using CacheValue = QVector<DbDataRecord>;
       
       // LRU缓存，最大容量100页
       QCache<CacheKey, CacheValue> m_cache{100};
       
       CacheValue get(const CacheKey& key);
       void put(const CacheKey& key, const CacheValue& value);
       void invalidate(int fileId);  // 文件数据变化时失效
       void clear();
   };

2. 预加载策略：
   - 当前页加载完成后，后台预加载相邻页（前一页+后一页）
   - 使用QtConcurrent::run在后台线程预加载
   - 预加载结果存入缓存

3. 缓存失效：
   - 筛选条件变化时清空缓存

   - 内存压力时自动清理最久未使用的缓存
```
