## 1. 项目概述

### 1.1 项目背景

1553B总线是航空及军工领域广泛使用的数据总线标准，具有高可靠性和实时性。本工具旨在提供一个智能化的1553B数据分析平台，通过连接大语言模型，实现对1553B数据的智能解析、查询和可视化分析。

### 1.2 项目目标

开发一款面向1553B数据的智能分析工具，支持以下核心功能：

- 二进制文件加载与1553B协议解析
- 表格展示
- 多大语言模型连接（千问、DeepSeek、Kimi、Ollama本地部署）
- 自然语言数据查询与分析
- 统计图表可视化（饼图、条形图、折线图等）
- 数据流动甘特图展示
- 数据导出功能（CSV、Excel、JSON、PDF、Word）

### 1.3 适用范围

- 1553B总线数据分析师
- 军工/航空电子设备研发人员
- 总线协议调试工程师

## 2. 数据结构定义

### 2.1 原始数据结构

本工具需支持解析以下数据结构：

```cpp
struct SMbiMonPacketHeader {
    quint16 header1;       // 固定值：A5 A5
    quint8 header2;         // 固定值：A5
    quint8 mpuProduceId;    // 任务机标识：1-mpu1; 2-mpu2
    quint16 packetLen;      // 包总长度
    quint16 year;           // 年
    quint8 month;           // 月
    quint8 day;             // 日
    quint32 timestamp;      // 时间戳 单位40us
};

struct CMD {
    quint16 zhongduandizhi : 5;  // 终端地址，bit 15-11
    quint16 T_R : 1;             // 收发状态，bit 10
    quint16 zidizhi : 5;         // 子地址，bit 9-5
    quint16 sjzjs_fsdm : 5;      // 数据计数/发送码，bit 4-0
};

struct SMbiMonPacketData {
    quint16 header;         // 固定值：AA BB
    CMD cmd1;                // 命令字1
    CMD cmd2;                // 命令字2
    quint16 states1;         // 状态字1
    quint16 states2;         // 状态字2
    quint16 chstt;           // 获取成功或失败
    quint32 timestamp;       // 时间戳 单位40us
    QByteArray datas;        // 数据区
};

struct SMbiMonPacketMsg {
    SMbiMonPacketHeader packetHeader;           // 包头
    QVector<SMbiMonPacketData> packetDatas;     // 数据数组
};
```

### 2.2 数据解析规则

| 规则 | 说明 |
|------|------|
| 包识别 | 通过header1=0xA5A5且header2=0xA5识别包起始 |
| 数据识别 | 通过header=0xAABB识别数据起始 |
| 包长度 | packetLen字段表示完整包的总长度 |
| 字节序 | 可通过XML配置，默认大端模式 |

### 2.3 命令字位域定义

CMD命令字为16位无符号整数，位域分布如下：

```
┌─────────────────────────────────────────────────────────────┐
│  bit 15-11  │  bit 10  │  bit 9-5  │  bit 4-0              │
│  终端地址    │  收发状态 │  子地址   │  数据计数/发送码       │
│  (5位)      │  (1位)   │  (5位)    │  (5位)                │
└─────────────────────────────────────────────────────────────┘
```

| 位域 | 字段名 | 位宽 | 取值范围 | 说明 |
|------|--------|------|----------|------|
| bit 15-11 | zhongduandizhi | 5位 | 0-31 | RT终端地址 |
| bit 10 | T_R | 1位 | 0/1 | 0=BC→RT, 1=RT→BC |
| bit 9-5 | zidizhi | 5位 | 0-31 | 子地址编号 |
| bit 4-0 | sjzjs_fsdm | 5位 | 0-31 | 数据字计数或发送模式码 |

### 2.4 数据区长度计算规则

SMbiMonPacketData.datas字节数根据cmd1字段动态计算：

| 条件 | datas长度 | 说明 |
|------|-----------|------|
| cmd1.zidizhi == 0 或 cmd1.zidizhi == 31 | 2字节 | 特殊子地址，固定2字节 |
| cmd1.sjzjs_fsdm == 0 | 64字节 | 数据计数为0时，固定64字节 |
| 其它情况 | cmd1.sjzjs_fsdm × 2 字节 | 数据计数×2（每字2字节） |

**计算公式（伪代码）：**

```
if (cmd1.zidizhi == 0 || cmd1.zidizhi == 31) {
    datasLength = 2;
} else if (cmd1.sjzjs_fsdm == 0) {
    datasLength = 64;
} else {
    datasLength = cmd1.sjzjs_fsdm * 2;
}
```

### 2.6 消息类型识别规则

根据命令字字段自动识别1553B消息类型：

| 消息类型 | 识别规则 | 说明 |
|----------|----------|------|
| BC→RT | cmd1.T_R=0，cmd2为空或无效 | 总线控制器发送数据到终端 |
| RT→BC | cmd1.T_R=1 | 终端发送数据到总线控制器 |
| RT→RT | cmd1和cmd2都有效，且地址不同 | 终端到终端的数据传输 |
| Broadcast | 终端地址=0x1F（31） | 广播消息，所有终端接收 |

**识别流程（伪代码）：**

```
if (cmd1.zhongduandizhi == 31) {
    messageType = "Broadcast";
} else if (cmd1.T_R == 0 && cmd2无效) {
    messageType = "BC→RT";
} else if (cmd1.T_R == 1) {
    messageType = "RT→BC";
} else if (cmd1有效 && cmd2有效 && 地址不同) {
    messageType = "RT→RT";
}
```

#### 2.7.1 性能目标

| 指标 | 当前性能 | 目标性能 | 提升倍数 |
|------|---------|---------|---------|
| 100万条数据导入 | 18分钟 | 10-30秒 | 36-108倍 |
| 筛选查询 | 数秒 | <100ms | 10-100倍 |
| 统计图表生成 | 1-2秒 | <200ms | 5-10倍 |
| 甘特图数据加载 | 2-3秒 | <500ms | 4-6倍 |

| 支持数据量 | 20万 | 1000万+ | 50倍+ |

#### 2.7.2 核心优化策略

**1. 并行导入**
- 多线程并行处理数据导入
- 每个线程处理独立的数据块
- 使用临时数据库避免锁竞争
- 最终合并到主数据库

**2. 批量插入**
- 使用批量VALUES语法减少SQL解析开销
- 每批插入1000条记录
- 禁用索引导入，导入后重建

**3. 列式存储**
- 为常用查询字段创建独立的列式存储表
- 优化分析型查询性能
- 支持快速统计和聚合操作

**4. 读写分离**
- 主数据库负责写入操作
- 多个只读副本负责查询操作
- 连接池管理数据库连接

DataStore类需要支持两种存储模式：

- 适用于小数据量（<10万条）
- 数据存储在内存中
- 快速访问和筛选

**数据库模式（Database Mode）：**
- 适用于大数据量（>=10万条）
- 数据存储在SQLite数据库中
- 支持并行导入和查询优化

**自动模式（Auto Mode）：**
- 根据数据量自动选择存储模式
- 阈值可配置（默认10万条）

**时间戳计算规则：**
- 原始timestamp单位为40微秒
- 转换为毫秒：timestamp * 40 / 1000
- 显示格式：HH:MM:SS.mmm
- 计算公式：
  - 总毫秒数 = timestamp * 40 / 1000
  - 小时 = 总毫秒数 / 3600000
  - 分钟 = (总毫秒数 % 3600000) / 60000
  - 秒 = (总毫秒数 % 60000) / 1000
  - 毫秒 = 总毫秒数 % 1000

### 3.10 日志系统

#### 3.10.1 日志功能需求

| 功能编号 | 功能名称 | 功能描述 | 优先级 |
|----------|----------|----------|--------|
| F-170 | 日志记录 | 记录应用程序运行过程中的关键操作和事件 | P0 |
| F-171 | 日志级别 | 支持DEBUG、INFO、WARNING、ERROR四个级别 | P0 |
| F-172 | 日志文件 | 日志自动写入文件，按日期命名 | P0 |
| F-173 | 日志轮转 | 支持日志文件大小限制和自动轮转 | P1 |
| F-174 | 模块标识 | 每条日志包含模块名称，便于追踪 | P0 |
| F-175 | 时间戳 | 每条日志包含精确到毫秒的时间戳 | P0 |

#### 3.10.2 日志配置参数

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| 日志文件路径 | 程序目录/logs/1553BAnalyzer_yyyyMMdd.log | 按日期自动命名 |
| 日志级别 | DEBUG | 可配置：DEBUG/INFO/WARNING/ERROR |
| 最大文件大小 | 10MB | 超过大小自动轮转 |
| 最大备份数量 | 5个 | 轮转后保留的历史文件数 |
| 编码格式 | UTF-8 | 日志文件编码 |

#### 3.10.3 日志级别说明

| 级别 | 标识 | 使用场景 |
|------|------|----------|
| DEBUG | DEBUG | 调试信息，详细的程序运行状态 |
| INFO | INFO | 一般信息，关键操作记录 |
| WARNING | WARN | 警告信息，非致命错误 |
| ERROR | ERROR | 错误信息，需要关注的异常 |

#### 3.10.4 日志格式

每条日志采用以下格式：

```
[时间戳] [级别] [模块名] 日志内容
```

示例：
```
[2024-03-31 10:30:45.123] [INFO ] [Main            ] 应用程序启动，版本: 1.0.0
[2024-03-31 10:30:45.156] [INFO ] [Main            ] Qt版本: 6.10.2
[2024-03-31 10:30:45.234] [DEBUG] [XmlConfigParser ] 开始解析配置文件: config/config.xml
[2024-03-31 10:30:45.345] [INFO ] [XmlConfigParser ] 配置文件解析完成，共加载 4 个模型提供商
[2024-03-31 10:30:46.789] [ERROR] [BinaryParser    ] 解析错误: 包头校验失败，位置: 0x1000
```

#### 3.10.5 主要日志记录点

| 模块 | 日志记录点 | 级别 |
|------|------------|------|
| Main | 应用程序启动/退出 | INFO |
| Main | Qt版本、工作目录 | INFO |
| ConfigManager | 配置文件加载成功/失败 | INFO/WARNING |

为方便日志记录，提供以下宏：

```cpp
#define LOG_DEBUG(module, message) Logger::instance()->debug(module, message)
```

```cpp
/**
```

```cpp
/**

 * - 命令字解析
```

```cpp
#define DATASTORE_H

#include <QVector>

 * @brief 数据记录结构

struct DataRecord {

    int msgIndex;              ///< 消息索引
    int dataIndex;             ///< 数据包索引
    SMbiMonPacketData packetData;  ///< 原始数据包

 * @brief 数据存储管理类

 * @code
 * DataStore store;
 * store.loadData(rawData);
 * QVector<DataRecord> records = store.getFilteredRecords();
 * @endcode
 */
class DataStore : public QObject {
    Q_OBJECT

public:
    /**
     * @brief 构造函数
     * @param parent 父对象指针
     */
    explicit DataStore(QObject* parent = nullptr);
    
    /**
     * @brief 加载原始数据
     * 
     * 将解析后的原始数据加载到存储中，
     * 并自动重建索引和计算统计信息
     * 
     * @param rawData 原始消息数据数组
     */
    void loadData(const QVector<SMbiMonPacketMsg>& rawData);
    
signals:
    /**

     * 
     * 当数据被重新加载或修改时发出
     */
```

**实现文件示例：**

```cpp
/**
 * @file DataStore.cpp
 * @brief 数据存储管理类实现
 */

#include "DataStore.h"

/**
 * @brief 构造函数
 * @param parent 父对象指针
 * 
 * 初始化数据存储对象，设置默认参数
 */
DataStore::DataStore(QObject* parent)
    : QObject(parent)
    , m_totalCount(0)
{
    // 初始化成员变量
}

/**
 * @brief 加载原始数据
 * 
 * 将解析后的原始数据加载到存储中，执行以下步骤：
 * 1. 清空现有数据
 * 2. 存储原始数据
 * 3. 重建索引
 * 4. 计算统计信息
 * 5. 发出数据变化信号
 * 
 * @param rawData 原始消息数据数组
 */
void DataStore::loadData(const QVector<SMbiMonPacketMsg>& rawData)
{
    // 清空现有数据
    m_rawData.clear();
    m_allRecords.clear();
    
    // 存储原始数据
    m_rawData = rawData;
    
    // 重建索引
    rebuildIndex();
    
    // 发出数据变化信号
    emit dataChanged();
}

/**
 * @brief 重建数据索引
 * 
 * 遍历所有原始数据，构建DataRecord数组，
 * 计算派生字段（如消息类型、时间戳等）
 */
void DataStore::rebuildIndex()
{
    m_allRecords.clear();
    int rowIndex = 0;
    
    // 遍历每条消息
    for (int msgIdx = 0; msgIdx < m_rawData.size(); ++msgIdx) {
        const SMbiMonPacketMsg& msg = m_rawData[msgIdx];
        
        // 遍历消息中的每个数据包
        for (int dataIdx = 0; dataIdx < msg.packetDatas.size(); ++dataIdx) {
            const SMbiMonPacketData& pkt = msg.packetDatas[dataIdx];
            
            // 创建数据记录
            DataRecord record;
            record.rowIndex = rowIndex++;
            record.msgIndex = msgIdx;
            record.dataIndex = dataIdx;
            record.packetData = pkt;
            record.packetHeader = msg.header;
            
            // 检测消息类型
            record.messageType = detectMessageType(pkt);
            
            // 计算时间戳（单位：毫秒）
            // 原始时间戳单位为40微秒，转换为毫秒需要乘以40/1000
            record.timestampMs = pkt.timestamp * 40.0 / 1000.0;
            
            m_allRecords.append(record);
        }
    }
}
```

#### 12.1 数据库结构优化

**新增字段：**
- `packets` 表新增 `msg_index` 字段，用于关联原始消息数据
- 通过 `msg_index` 可直接从内存中的 `m_rawData` 获取完整的消息头信息

**删除内容：**
- 移除 `messages` 表（不再需要单独存储消息头信息）
- 移除 `packets` 表对 `messages` 表的外键引用

**优化说明：**
消息头信息（`SMbiMonPacketHeader`）存储在内存的 `m_rawData` 中，通过 `msg_index` 索引访问，避免了数据冗余存储，简化了数据库结构。

**版本管理：**
- 数据库版本号存储在 `db_version` 表中
- 当前版本：V2
- 支持自动检测并执行升级脚本

2. 并行导入到临时数据库，存储 `msg_index` 和数据包信息
3. 合并临时数据库到主数据库
4. 保留原始数据在内存中（`m_rawData`）

**查询流程：**
1. 从数据库查询 `DbDataRecord`，获取 `msg_index` 和 `data_index`
2. 通过 `msg_index` 从 `m_rawData` 获取消息头信息
3. 通过 `data_index` 获取具体的数据包内容
4. 组装成完整的 `DataRecord` 返回给UI
