# 第二部分：隐藏Bug分析

## 2.1 【严重】DataStore::applyFiltersAsync 线程安全问题

**位置**：[DataStore.cpp:1608-1690](file:///d:/00-code/06-1553BNew/src/core/datastore/DataStore.cpp#L1608)

**问题描述**：
```cpp
void DataStore::applyFiltersAsync()
{
    m_cancelFilter = false;
    int total = m_allRecords.size();  // ① 主线程读取
    QVector<DataRecord> allRecordsCopy = m_allRecords;  // ② 主线程拷贝
    
    QtConcurrent::run([this, total, allRecordsCopy, sortField, sortOrder]() {
        // ③ 后台线程使用 allRecordsCopy（安全）
        // ④ 但 matchesFilter() 读取 m_terminalFilter 等成员变量（无锁）
        for (const DataRecord& record : allRecordsCopy) {
            if (m_cancelFilter) { ... }
            if (matchesFilter(record)) { ... }  // 读取主线程可能同时修改的筛选条件
        }
        
        QMetaObject::invokeMethod(this, [this, filtered]() {
            m_filteredRecords = filtered;  // ⑤ 主线程写入（安全）
        }, Qt::QueuedConnection);
    });
}
```

**风险**：
- `matchesFilter()` 读取 `m_terminalFilter`、`m_subAddressFilter` 等成员变量，这些变量可能在主线程被同时修改
- `m_cancelFilter` 是 `std::atomic<bool>` 但筛选条件成员不是原子变量
- 如果用户在异步筛选进行中修改筛选条件，可能导致数据不一致或崩溃

**修复建议**：在拷贝 `allRecordsCopy` 时同时拷贝所有筛选条件，在lambda中使用拷贝而非成员变量。

## 2.2 【严重】DataStore 同步/异步筛选竞态

**位置**：[DataStore.cpp:629](file:///d:/00-code/06-1553BNew/src/core/datastore/DataStore.cpp#L629) 与 [DataStore.cpp:1608](file:///d:/00-code/06-1553BNew/src/core/datastore/DataStore.cpp#L1608)

**问题描述**：
- `applyFilters()` 是同步方法，在主线程直接修改 `m_filteredRecords`
- `applyFiltersAsync()` 是异步方法，在后台线程筛选后通过 `QMetaObject::invokeMethod` 写入 `m_filteredRecords`
- 如果异步筛选正在进行时，用户操作触发了同步的 `applyFilters()`，两者会同时操作 `m_filteredRecords`
- `setIndexedData()` 中（L1462）先检查筛选条件，有则调用 `applyFiltersAsync()`，无则直接赋值 `m_filteredRecords`，若此时有异步筛选正在进行，直接赋值可能覆盖异步筛选的结果

**修复建议**：在 `applyFilters()` 开头调用 `cancelAsyncFilter()` 并等待异步操作完成；或使用 `QFuture` 跟踪异步操作状态。

## 2.3 【中等】MainWindow::generateCmd2 完全冗余

**位置**：[MainWindow.cpp:2388-2390](file:///d:/00-code/06-1553BNew/src/ui/mainwindow/MainWindow.cpp#L2388)

虽然当前逻辑上cmd1和cmd2的构建方式相同，但1553B协议中cmd2是响应方的命令字，其T/R位含义与cmd1相反。当前实现中调用时已手动交换了isSend参数（L2250: `cmd2 = generateCmd2(terminalAddr2, subAddr2, true, dataCount1)`），但函数名和参数名具有误导性，容易在后续维护中引入bug。

**修复建议**：删除 `generateCmd2()`，直接调用 `generateCmd1()` 并在调用处明确注释T/R位的含义。

# 第三部分：优化建议

## 3.1 DataStore 筛选性能优化 - 提前终止与索引

**位置**：[DataStore.cpp:712-869](file:///d:/00-code/06-1553BNew/src/core/datastore/DataStore.cpp#L712)

**现状**：`matchesFilter()` 对每条记录执行10+个条件判断，且每次都通过 `memcpy` 解析CMD结构体。

**优化建议**：
1. **CMD解析缓存**：在 `DataRecord` 中缓存已解析的 `terminalAddr`、`subAddr`、`dataCount` 等字段，避免每次筛选都重新解析
2. **筛选条件排序**：将选择性最高的条件（如终端地址）放在最前面，利用短路求值提前跳过不匹配的记录
3. **位运算替代memcpy**：直接使用位运算提取CMD字段，比memcpy+结构体访问更快：
   ```cpp
   int terminalAddr = (cmd1Raw >> 11) & 0x1F;
   int subAddr = (cmd1Raw >> 5) & 0x1F;
   int dataCount = cmd1Raw & 0x1F;
   ```

## 3.2 ReportGenerator 图表生成优化 - 延迟渲染

**位置**：[ReportGenerator.cpp](file:///d:/00-code/06-1553BNew/src/report/ReportGenerator.cpp)

**现状**：报告生成时会一次性生成5张图表（饼图、柱状图、折线图、错误分布图、成功率图），每张都渲染为QImage，占用大量内存。

**优化建议**：
1. **按需渲染**：仅在需要导出时才渲染图表，预览模式使用缩略图
2. **分辨率分级**：PDF导出使用高分辨率（当前2x），HTML预览使用1x分辨率
3. **图表复用**：如果用户连续生成报告，可复用未变化的图表

## 3.3 DatabaseManager 查询缓存优化

**位置**：[DatabaseManager.cpp](file:///d:/00-code/06-1553BNew/src/core/datastore/DatabaseManager.cpp)

**现状**：每次分页查询都执行完整的SQL查询，无缓存机制。

**优化建议**：
1. **查询结果缓存**：对相同的筛选条件和页码，缓存查询结果
2. **预加载下一页**：在用户浏览当前页时，后台预加载下一页数据
3. **COUNT查询优化**：缓存 `queryPacketCount` 的结果，避免每次翻页都执行COUNT(*)

## 3.4 AI Provider 网络请求优化

**位置**：所有Provider的 `chat()` 同步方法

**现状**：每个Provider的 `chat()` 方法都创建局部的 `QNetworkAccessManager`，这意味着每次同步调用都创建和销毁网络管理器，无法复用TCP连接。

**优化建议**：使用共享的 `QNetworkAccessManager` 实例，利用HTTP Keep-Alive复用连接，减少延迟。

## 3.5 内存使用优化 - 数据分片加载

**位置**：[DataStore.cpp](file:///d:/00-code/06-1553BNew/src/core/datastore/DataStore.cpp)

**现状**：内存模式下，`m_allRecords` 和 `m_filteredRecords` 各保存一份数据副本，对于大数据集（>50万条）内存占用翻倍。

**优化建议**：
1. **索引分离**：`m_filteredRecords` 仅保存索引（int），不保存完整数据副本
2. **懒加载**：仅在需要时从数据库或原始数据中加载完整记录
3. **数据压缩**：对不常用的字段（如datas字段）使用压缩存储

## 3.6 UI响应优化 - 批量信号合并

**位置**：[DataStore.cpp](file:///d:/00-code/06-1553BNew/src/core/datastore/DataStore.cpp)

**现状**：筛选条件变更时，每个 `setXxxFilter()` 都调用 `applyFilters()`，如果用户批量修改多个筛选条件，会触发多次全量筛选。

**优化建议**：虽然已有 `beginBatchFilterUpdate()` / `endBatchFilterUpdate()` 机制，但当前仅在一处使用。建议在所有AI工具执行筛选时都使用批量更新，减少不必要的重复筛选。
